Context

What is the Cloud?

Computing and software resources that are delivered on demand, as service. (A Walk in the Clouds. Cloud Computing, CDW-G Reference Guide., 3-5)

Cloud Resources Management

Management of set of heterogeneous Resources :
  • Different software versions, VM Management Platforms
  • Different physical machines characteristics
  • Concurrent execution of multiple operating systems
Need for Resources Placement Strategies :
  • Migrate VMs to more utilized nodes, Shutdown unused nodes
  • Schedule VMs to physical nodes, Virtual Machines isolation

Resources Placement Problem

Motivation

Abstraction for Elasticity Management

“the ability of software to meet changing capacity demands, deploying and releasing necessary resources on-demand”[Yefim et al.]

Need to provision/deprovision resources dynamically as response to:
  • Increasing Workloads, Incident Response
  • Changes in SLA requirements
We need an abstraction layerand a reasoning engine to reason about cloud adaptations

Conflicting Objectives in VM Management

How to place VM hosting applications belonging to many customers?

Cloud Management

  • Cloud management (i.e., VM elastic management) has to deal with trade-offs such as cost, quality of service and security.
  • Cloud Management is a multi-objective optimization problem
  • Current cloud platforms usually focus on one dimensional optimization

Motivation

Writing a reasoner to handle these trade-offs is difficult:

  • Several conflicting optimization axis
  • Inputs in each axis are continuously evolving
  • Resources are heterogeneous

Assumption: A cloud is a DAAS

The Cloud is a Dynamic Adaptive system:

  • Reconfiguration capabilities at runtime, to adjust to unpredictable conditions: reflection
  • Capacity of observing their own behavior and analyzing their effectiveness: introspection
  • Ability to change their behavior: intercession

Models at run.time for the cloud

Resolution

Problem Inputs

  • A set of requirements that we must satisfy over time: objectives
  • A cloud configuration: a model
  • A set of actions to update a configuration: operators

Resolution Approaches

  • Constraint Solving Techniques
  • Simulated Annealing
  • Tabu Search
  • We advocate MOEA (Multi-objective Evolutionary Algorithm):
  • Population based
  • Possible optimal solutions at any generation

Overall approach

CMOP Inputs

CMOP Modeling

Models@run.time for Cloud abstraction:

  1. Provide high level abstraction for Clouds through a standard MOF-based metamodel
  2. Enable to reason about the system at design time
  3. Extend the reasoning to the system while it is in a running state
  4. Reevaluate continuously requirements satisfaction while the system is evolving.

CMOP Modeling

  1. A physical machine → Kevoree Infrastructure Node
  2. A virtual machine → Kevoree Child Node
  3. A component → Software service

CMOP Problem Domain Inputs

We define CMOP by the following triplet (I,F,CO) such that:

  1. I denotes a cloud infrastructure model.
  2. F is a vector of objective functions F(X) = ( f1(x),f2(x),...,fn(X)), that have to be minimized.
  3. CO denote possible cloud configurations: co = VM1(c1,c2,c3) denotes a configuration with a single virtual machine VM1 hosting 3 components c1,c2,c3.

Operators Description

Operators Description
AddComponent(c,A) Adds a component c in the the virtual machine A
RemoveComponent(c,A) Removes a component c in the virtual machine A
MoveComponent(c,A,B) Moves a component c from the virtual machine A to the virtual machine B

Fitness Functions

  1. f1(x)= Cost(x) calculated based on Amazon pricing model (http://aws.amazon.com/ec2/pricing/)
  2. f2(x)=Security(x) defines the security level associated with a Component
  3. f3(x)=Completeness(x) provides a score of the number of software components that have been placed
  4. f4(x)=Overload(x) defines the gap between the required and the observed % of virtual CPU for each software element
  5. ......

Validation

Validation Objectives

Validation of our approach is related to the scenarios of interest for EBRC provider in Luxembourg and aims at:

  1. Providing a supervision approach to reconfigure a cloud platform according to evolving customer optimization objectives
  2. Adjusting customers resources provisioning and quality of service to the deployment costs
  3. Reducing data centers energy consumption at the level of the cloud provider

Experimental Setup

EBRC allocates to one of its customers a cluster composed of 5 physical machines to host its on-line stock trading website that is composed of the following components:

  1. A database to store items
  2. A load balancer
  3. A database for payment
  4. A database to manage users
  5. A web Front-end

Experimental Setup

5 physical machines that we deploy in Kevoree:

  1. 3 ARM based Infrastructure Node (1 Virtual Machine abstracted as child node where each node is 1GHz)
  2. 2 Xeon based Infrastructure Node (8 Virtual Machine abstracted as child node where each node is 1GHz)

Validation Objectives

We aim at investigating the Feasibility and the Scalabilityof our approach:

  1. Comparing mono-objective vs. multi-objective optimization
  2. Comparing multi-objective algorithms in terms of objectives satisfaction and scalability

Research Questions

  1. What is the benefit of using multi-objective optimization compared to mono-objective one? How faster a good solution is found using a mono-objective?
  2. Which algorithm: CMOP-epsilon Dominance, CMOP without Epsilon Dominance and Random algorithm better satisfies the objectives while scaling to more complex cloud infrastructure model?

Threats to Validity

Internal Validity:
  1. Number of generations and populations at the initial setup
  2. Stopping criterion
  3. Fitness functions models
External validity:
  1. Pertinence of the chosen optimization axis
  2. Selected resolution strategies

Conclusion

Conclusion

  1. Effectiveness of the optimization at the level of architectural models: reasonable solutions in reasonable time, acceptable in the case of cloud dynamic reconfiguration
  2. Effectiveness of search-based approaches to resolve a cloud multi-objective optimization problem
  3. Implementation using Kevoree as a Models@run.time platform

Future Work

  1. Detection of the absence of improvements as a stopping criterion
  2. Comparison with other resolution strategies
  3. Usage of Hyper-heuristics to improve the search performance
